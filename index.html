MODULE 01 · OBC: THE BRAIN OF THE SATELLITE

BASICS · 15 MIN

WHAT IS AN OBC?

The Onboard Computer (OBC) is the central processing unit of a spacecraft. Think of it as the satellite's brain — it coordinates every other subsystem, executes the mission logic, handles data, and keeps everything running reliably in the harshest environment imaginable: outer space.

Just like your laptop has a CPU, RAM, and storage — so does an OBC. But it's designed for extreme temperatures (-40°C to +85°C), vacuum, and most critically, radiation from cosmic rays and the Van Allen belts.

CORE RESPONSIBILITIES

  • Command Execution — receives telecommands from ground station and routes them to the right subsystem
  • Telemetry Collection — gathers health data (temperature, voltage, current) from all subsystems
  • Data Handling (C&DH) — stores, compresses, and downlinks mission/science data
  • Attitude Control Interface — sends/receives data to/from ADCS (Attitude Determination & Control)
  • Power Management Interface — monitors and controls EPS (Electrical Power System)
  • Fault Management — detects anomalies and executes recovery procedures
  • Time Keeping — maintains onboard time reference for scheduling and synchronization

MODULE 02 · HARDWARE COMPONENTS

BASICS · 15 MIN

THE PROCESSOR (CPU)

Space OBCs typically use ARM Cortex-M or LEON3/LEON4 (SPARC-based, ESA's standard) processors. For student CubeSats, ARM Cortex-M4/M7 is extremely common because of its low power, good performance, and wide ecosystem.

MEMORY TYPES — CRITICAL TO KNOW

  • SRAM (Static RAM) — Fast volatile memory for runtime data. 256KB–8MB typical. Loses data on power loss.
  • Flash Memory (NOR/NAND) — Non-volatile. Stores the flight software and science data. NOR for code execution, NAND for bulk data storage.
  • EEPROM — Non-volatile, byte-writable. Stores mission-critical config parameters that must survive resets.
  • eDRAM / SDRAM — High-density RAM for data-intensive payloads like cameras. Needs refresh cycles.

WATCHDOG TIMER

The watchdog timer (WDT) is one of the most important safety features. The flight software must "kick" (reset) the watchdog periodically. If the software hangs or crashes, the watchdog times out and forces a hardware reset, rebooting the OBC automatically — without any human intervention from Earth.

// Pseudo-code: Watchdog pattern while (satellite_running) { execute_tasks(); kick_watchdog(); // Must happen every ~1–30 sec // If this line is never reached → WDT resets CPU }

RADIATION HARDENING

Normal COTS (commercial off-the-shelf) chips can suffer Single Event Upsets (SEUs) — a cosmic ray flips a bit from 0 to 1. Space-grade chips use:

  • Radiation Hardened (RadHard) ICs — physically designed to resist radiation. Expensive ($$$)
  • Radiation Tolerant (RadTol) COTS — selected COTS parts tested for space use. More affordable for student teams
  • Error Correcting Code (ECC) Memory — detects and corrects single-bit errors in RAM
⚠ SPACE CLUB NOTE

For a student CubeSat in Low Earth Orbit (LEO), radiation environment is less severe than deep space. Using a well-chosen ARM microcontroller with software fault tolerance is acceptable and very common.

MODULE 03 · COMMUNICATION INTERFACES

INTERMEDIATE · 15 MIN

WHY INTERFACES MATTER

The OBC needs to talk to ADCS, EPS, sensors, the radio, and payload. Each uses a specific protocol. Knowing which protocol to use for what is fundamental to OBC design.

THE FOUR KEY PROTOCOLS

UART — UNIVERSAL ASYNC RECEIVER/TRANSMITTER

Simple 2-wire serial (TX/RX). Point-to-point only. Used for: GPS modules, debug console, radio comms module. Speed: 9600 bps to ~1 Mbps. Easy to implement — the "hello world" of interfaces.

SPI — SERIAL PERIPHERAL INTERFACE

4-wire (MOSI, MISO, CLK, CS). Full duplex, master-slave. Used for: Flash memory, ADCs, sensors. Very fast (10s of Mbps). Requires one CS pin per device — can get complex with many devices.

I2C — INTER-INTEGRATED CIRCUIT

2-wire (SDA, SCL). Multi-master, multi-slave. Used for: temperature sensors, IMUs, EPS monitoring. Slower (400 KHz typical) but super efficient for many sensors on one bus. Each device has a unique address.

CAN BUS — CONTROLLER AREA NETWORK

Differential 2-wire bus. Highly reliable, built-in error detection. Borrowed from automotive. Used in: higher-end CubeSats for robust inter-subsystem comms. Speed: 1 Mbps. Excellent fault tolerance — can handle wire faults.

QUICK COMPARISON TABLE

Protocol | Wires | Speed | Distance | Use Case ---------|-------|-----------|----------|------------------ UART | 2 | ~1 Mbps | Short | Serial debug, GPS SPI | 4+ | 10+ Mbps | Short | Flash, fast sensors I2C | 2 | 400 KHz | Short | Many sensors, EPS CAN Bus | 2 | 1 Mbps | Medium | Robust subsystem bus RS-422 | 4 | 10 Mbps | Long | Long cable runs
✓ PRACTICAL TIP

Most student CubeSat OBCs use I2C for sensors and EPS, SPI for memory and fast peripherals, and UART for the radio module and debugging. CAN Bus appears in more serious teams or commercial boards.

MODULE 04 · RTOS & FLIGHT SOFTWARE

INTERMEDIATE · 15 MIN

WHAT IS AN RTOS?

A Real-Time Operating System (RTOS) guarantees that critical tasks complete within a defined time window. In space, missing a deadline can mean losing attitude control or missing a downlink window. Unlike your phone's OS, an RTOS prioritizes determinism over features.

TASK SCHEDULING

Flight software is split into tasks (threads), each with a priority and timing requirement. The RTOS scheduler runs the highest-priority ready task. Common OBC tasks:

  • Attitude Control Task — highest priority, runs every 100ms (time-critical!)
  • Telemetry Collection Task — runs every 1 second, collects housekeeping data
  • Command Handler Task — runs whenever a new telecommand arrives
  • Health Monitor Task — checks subsystem health, triggers FDIR if needed
  • Data Storage Task — writes science/mission data to flash memory
  • Watchdog Kick Task — must run within watchdog period or system resets

SATELLITE STATE MACHINE

The OBC software implements a Finite State Machine (FSM) to manage mission modes. This ensures the satellite behaves predictably in every scenario:

LAUNCH_WAIT → DEPLOYMENT_DELAY → SAFE_MODE ↓ NOMINAL_OPS ←→ SCIENCE_MODE ↓ EMERGENCY_MODE → RECOVERY
FLIGHT SOFTWARE LAYERS

Hardware Abstraction Layer (HAL) → Board Support Package (BSP) → RTOS Kernel → Device Drivers → Application Layer (Mission Logic). You'll likely work at the Application Layer and Drivers level for your OBC.

MODULE 05 · FAULT TOLERANCE & FDIR

ADVANCED · 15 MIN

THE SPACE ENVIRONMENT IS HOSTILE

In space, you cannot send a technician to fix a problem. The OBC must detect, isolate, and recover from faults autonomously. This is called FDIR — Fault Detection, Isolation, and Recovery.

RADIATION EFFECTS ON ELECTRONICS

  • SEU (Single Event Upset) — a cosmic particle flips a single bit in memory. Most common effect.
  • SEL (Single Event Latchup) — causes a short circuit in CMOS chip. Can permanently destroy the IC if not caught.
  • TID (Total Ionizing Dose) — cumulative radiation degrades transistors over mission lifetime.
  • SEE (Single Event Effects) — umbrella term for all radiation-induced transient faults.

FAULT TOLERANCE TECHNIQUES

TRIPLE MODULAR REDUNDANCY (TMR)

Three identical circuits perform the same computation. A majority voter selects the output — if one fails (flips a bit), the other two outvote it. Used in critical data paths and memory. "Two out of three wins."

ERROR CORRECTING CODE (ECC) MEMORY

Hardware automatically detects and corrects single-bit errors in RAM. Can detect double-bit errors. Essential for any space RAM. SECDED (Single Error Correct, Double Error Detect) is the most common scheme.

WATCHDOG TIMER (WDT)

Software must "kick" the watchdog within a timeout window. If software hangs, WDT forces a hard reset. First line of defense against software crashes in space.

SAFE MODE

When too many faults are detected, OBC transitions to Safe Mode: stops non-essential operations, points solar panels at the Sun, and awaits ground commands. Think of it as the satellite "calling for help."

FDIR DECISION TREE

Anomaly Detected ↓ Is it recoverable? ├── YES → Attempt automated recovery │ ↓ (recovery fails 3x) │ Escalate to Safe Mode └── NO → Immediately enter Safe Mode ↓ Transmit error telemetry to ground ↓ Await ground command to resume

MODULE 06 · OBC IN THE FULL SATELLITE SYSTEM

ADVANCED · 15 MIN

COMMAND & DATA HANDLING (C&DH)

The OBC IS the C&DH subsystem. Its job is to manage all information flow on the satellite. C&DH sits at the center of every satellite architecture.

  • Telecommand (TC) — commands coming up from ground to the satellite (uplink)
  • Telemetry (TM) — data going down from satellite to ground (downlink) — housekeeping + science data
  • Packet Protocol — CCSDS (Consultative Committee for Space Data Systems) is the standard. Defines how packets are structured for TC and TM.

INTERFACE WITH EACH SUBSYSTEM

OBC ↔ ADCS (Attitude Determination & Control)

OBC sends attitude commands (desired orientation). ADCS sends back current attitude data (quaternions, angular rates). Typically uses SPI or I2C or CAN Bus. Time-critical — usually the highest-frequency interface.

OBC ↔ EPS (Electrical Power System)

OBC monitors battery voltage, solar panel current, power consumption of each subsystem. Can command EPS to switch power rails on/off (e.g. turn off payload during low power). Uses I2C typically.

OBC ↔ COMMS (Radio)

OBC passes telecommand packets received from the radio to the command handler. Prepares telemetry packets and sends to radio for downlink. Uses UART or SPI.

OBC ↔ PAYLOAD

OBC schedules payload operations (e.g. "start camera at 14:32:00 UTC"). Receives payload data and stores in onboard mass memory. May compress data before downlink.

DATA FLOW IN A PASS

Satellite enters ground station visibility window ↓ Radio receives telecommand packet ↓ OBC decodes command → routes to target subsystem ↓ Subsystem executes, returns telemetry ↓ OBC formats telemetry into CCSDS packets ↓ Radio downlinks packets to ground station ↓ Satellite passes out of range → wait for next pass

MODULE 07 · REAL OBC EXAMPLES

APPLIED · 10 MIN

COMMERCIAL CUBESAT OBCs

GOMSPACE NPC-OBC — INDUSTRY STANDARD

Based on AVR32 microcontroller. Has built-in CAN Bus, I2C, SPI, UART. FreeRTOS. Used in hundreds of CubeSats. Runs the CSP (CubeSat Space Protocol) which is an open networking protocol for CubeSats. Very well documented — good reference for your team.

ISIS OBC — POPULAR IN STUDENT MISSIONS

ARM7 based (AT91SAM9G20). SD card for mass storage, multiple interfaces. Used by many universities. ISIS (Innovative Solutions In Space) is a leading nano-satellite company.

ENDUROSAT OBC — MODERN ARM CORTEX-M7

High-performance, radiation-tolerant. 216 MHz processor, 2MB SRAM, 8MB Flash. Popular in 2020s student and commercial missions.

STUDENT / DIY OBC OPTIONS

  • STM32F4/F7 — ARM Cortex-M4/M7. Very popular for student OBCs. Free, excellent ecosystem. Run FreeRTOS easily.
  • Raspberry Pi + Microcontroller — Some teams use RPi for high-level logic + a microcontroller for real-time tasks. Power-hungry but powerful.
  • Arduino-based — Simple educational builds. Not suitable for real missions but great for learning.
  • FPGA-based OBCs — Advanced teams use Xilinx/Intel FPGAs for radiation tolerance + high performance. Steep learning curve.

KNOWLEDGE CHECK — 10 QUESTIONS
1. What does OBC stand for and what is its primary role in a satellite?
The OBC is the Onboard Computer — the satellite's brain. It handles command execution, telemetry, data management, and interfaces with every other subsystem.
2. What is a Watchdog Timer used for?
The watchdog timer must be "kicked" (reset) periodically by the flight software. If it isn't kicked (because software crashed), the WDT forces a hardware reset — an autonomous recovery mechanism critical for space.
3. Which communication interface uses only 2 wires and supports multiple devices with unique addresses on the same bus?
I2C (Inter-Integrated Circuit) uses just SDA and SCL wires, supports a multi-device bus where each device has a unique 7-bit address. Perfect for many sensors on a single bus.
4. What is an SEU in the context of space electronics?
SEU (Single Event Upset) occurs when a cosmic ray or energetic particle strikes a memory cell and flips its bit from 0 to 1 or vice versa. ECC memory can detect and correct these.
5. What does FDIR stand for?
FDIR — Fault Detection, Isolation, and Recovery — is the autonomous system by which the OBC identifies a problem, isolates it from the rest of the satellite, and recovers without human intervention.
6. What type of memory is used to store flight software code that must survive power cycles?
NOR Flash is non-volatile (survives power loss) and allows in-place code execution (XIP — Execute In Place). It's the standard choice for storing bootloaders and flight software on OBCs.
7. What does "Safe Mode" mean for a satellite?
Safe Mode is a minimal survival state — the satellite stops non-critical operations, orients for maximum power, and transmits beacons awaiting recovery commands. The satellite's last resort before total loss.
8. In Triple Modular Redundancy (TMR), how is the correct output determined?
TMR uses three identical circuits and a majority voter. If one flips a bit due to radiation, the other two "outvote" it. This masks single-point failures completely and is critical for radiation-tolerant design.
9. What is the CCSDS standard used for in satellite communication?
CCSDS (Consultative Committee for Space Data Systems) defines standardized packet structures for Telecommand (TC) and Telemetry (TM). Using CCSDS allows compatibility with ground stations worldwide.
10. Which interface protocol is most recommended for a robust inter-subsystem bus that handles electrical faults well?
CAN Bus (Controller Area Network) has built-in error detection, differential signaling (resistant to noise), and can handle bus faults. Originally designed for automotive reliability — ideal for the subsystem bus on more serious CubeSats.
